1. Executive Summary

OpenDeepResearcher is a sophisticated AI-powered research assistant that combines advanced large language models with a multi-agent orchestration framework (LangGraph) to conduct comprehensive web-based research. The system delivers a modern, ChatGPT-style web interface that seamlessly integrates intelligent research capabilities with persistent session memory for context-aware interactions.

The application represents a production-quality solution for researchers, students, and professionals who need deep, structured research capabilities with an intuitive user experience. It has reached Milestone 3, which includes a fully functional UI, session memory integration, and a robust FastAPI backend.

Key Highlights:

  • Multi-agent AI pipeline with intelligent task decomposition
  • Real-time web search integration using Tavily API
  • Persistent conversation history and session management
  • Modern, responsive ChatGPT-inspired interface
  • Full markdown support for formatted research reports
  • Production-ready FastAPI backend with database integration

2. Project Overview

2.1 Problem Statement

Traditional research processes are time-consuming and often fragmented. Researchers must manually:

  • Formulate comprehensive search queries across multiple sources
  • Aggregate and synthesize information from disparate results
  • Maintain context across multiple research sessions
  • Organize findings into coherent, formatted reports
  • Deal with information overload and relevance filtering

2.2 Solution Overview

OpenDeepResearcher automates and intelligently orchestrates this research workflow using a sophisticated multi-agent pipeline:

Research Pipeline Architecture:

Query → Decomposition → Search → Synthesis → Report Generation

  • Planner Agent: Breaks down complex queries into 4-6 strategic sub-questions
  • Searcher Agent: Conducts parallel web searches for each sub-question using Tavily API
  • Writer Agent: Synthesizes search results into comprehensive, well-structured reports
  • Memory System: Maintains conversation context for multi-turn interactions

2.3 Core Value Propositions

  1. Intelligent Research Automation: AI-driven decomposition of complex research queries reduces manual effort by 70%+
  2. Comprehensive Results: Multi-angle approach ensures thorough coverage of research topics
  3. Context Awareness: Session memory enables follow-up questions and iterative refinement
  4. Professional Presentation: Markdown-formatted reports ready for immediate use or further editing
  5. Scalable Architecture: LangGraph-based design allows easy addition of new agents and capabilities

4.4 Data Flow & Processing Pipeline

Step 1: User Query Reception

User submits research query via ChatInput component → HTTP POST to /research endpoint with session ID

Step 2: Session Management

Backend validates/creates session → Retrieves last 3 conversation exchanges from database → Injects context into new query

Step 3: Query Planning

Planner Agent processes topic with GPT-4 → Generates 4-6 strategic sub-questions → Returns structured question list

Step 4: Parallel Search

Searcher Agent iterates through sub-questions → Calls Tavily API for each question → Aggregates search results with metadata

Step 5: Report Synthesis

Writer Agent processes search results with GPT-4 → Generates comprehensive, well-formatted research report → Returns markdown-formatted response

Step 6: Storage & Response

Store user message and AI response in database → Return JSON response to frontend → Frontend renders message bubbles with markdown support

5. Multi-Agent Research Pipeline

5.1 Planner Agent

Purpose:

Intelligent query decomposition and strategic research question generation

Capabilities:

  • Breaks complex research topics into 4-6 focused sub-questions
  • Ensures diverse angles covering analysis, comparison, impacts, and challenges
  • Avoids duplicate or generic questions
  • Uses GPT-4 with temperature 0.2 for consistency

Input:

Research topic (string)

Output:

List of strategic sub-questions

Prompt Strategy:

System prompt defines role as "senior research strategist" with explicit guidelines for question quality and format

5.2 Searcher Agent

Purpose:

Real-time web search across multiple sub-questions

Capabilities:

  • Processes multiple search queries in parallel
  • Integrates with Tavily API for premium search results
  • Returns structured results with source attribution
  • Handles search failures gracefully

Input:

List of sub-questions from Planner

Output:

Aggregated search results with source URLs, snippets, and relevance metadata

5.3 Writer Agent

Purpose:

Synthesis and report generation from search results

Capabilities:

  • Synthesizes multiple search results into coherent narrative
  • Generates markdown-formatted reports with proper structure
  • Maintains source attribution throughout report
  • Uses GPT-4 for high-quality content generation

Input:

Aggregated search results from Searcher

Output:

Comprehensive markdown research report

Report Structure:

  • Executive Summary
  • Key Findings (organized by topic)
  • Analysis & Insights
  • Source Citations

5.4 LangGraph Workflow

Pipeline: Planner → Searcher → Writer (Linear execution) State Management: ResearchState TypedDict - topic: str (research topic/query) - sub_questions: List[str] (from planner) - search_results: List[Dict] (from searcher) - summary: str (from writer) Graph Structure: StateGraph with 3 nodes - Entry Point: Planner Node - Edges: planner → searcher → writer → END - No loops or cycles (linear pipeline)

6. Database Schema & Storage

6.1 Database Overview

Type: SQLite (lightweight, file-based)

Location: backend/research.db

Purpose: Persistent storage of conversations, messages, and session metadata

6.2 Core Tables

Table: conversations

Column Type Description
id INTEGER PRIMARY KEY Unique conversation identifier
session_id TEXT UNIQUE External session identifier (UUID)
title TEXT Auto-generated or user-provided conversation title
created_at TIMESTAMP Conversation creation timestamp
updated_at TIMESTAMP Last message timestamp

Table: messages

Column Type Description
id INTEGER PRIMARY KEY Unique message identifier
conversation_id INTEGER FOREIGN KEY Reference to conversations table
role TEXT 'user' or 'assistant'
content TEXT Message content (supports markdown)
created_at TIMESTAMP Message creation timestamp

6.3 Session Memory Cache

In addition to database storage, the backend maintains an in-memory session cache for performance:

  • Thread-safe dictionary-based cache
  • Stores recent messages for fast access
  • Automatically synced with database on write operations
  • Session validation and lifecycle management

13. Known Limitations & Technical Considerations

13.1 Current Limitations

  • Single-threaded Processing: Sequential pipeline execution (no parallelization within agents)
  • SQLite Database: Not suitable for concurrent write operations at scale
  • In-memory Session Cache: Lost on server restart without persistence
  • No Authentication: Current version suitable for single-user/trusted environment only
  • Web Search Only: Limited to public web content (no academic databases yet)
  • Fixed Agent Count: 3-node pipeline (Planner → Searcher → Writer)

13.2 API Key Dependency

The system depends on external paid APIs:

  • OpenAI GPT-4 API (~$0.03-0.06 per research query)
  • Tavily Search API (~$0.50-1.00 per research query)

13.3 Quality Considerations

  • Search result quality depends on Tavily API performance
  • Report synthesis quality depends on GPT-4 capabilities and prompting
  • No built-in fact-checking or verification mechanisms
  • Long context windows may reduce coherence in very long conversations

14. Security & Privacy Considerations

14.1 Current Security Posture

  • CORS Restriction: Limited to localhost ports (development only)
  • No Authentication: Session IDs are UUIDs but not cryptographically secure
  • Environment Variables: API keys stored locally in .env file
  • Database: SQLite file stored locally with no encryption

14.2 Production Security Requirements

  • Implement OAuth 2.0 or JWT-based authentication
  • Use HTTPS/TLS for all network communication
  • Store API keys in secure vault (AWS Secrets Manager, HashiCorp Vault)
  • Encrypt sensitive data at rest (database encryption)
  • Implement rate limiting and DDoS protection
  • Add input validation and sanitization
  • Regular security audits and penetration testing

14.3 Data Privacy

  • Conversation data is shared with external APIs (OpenAI, Tavily)
  • Implement data retention policies for GDPR/CCPA compliance
  • Add user data export and deletion functionality
  • Document privacy practices and data handling procedures
